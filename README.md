# Phaeton ETL Loader

Node.js-загрузчик ежедневных Excel-выгрузок из 1С (прайс-лист, скидки, ДЗ,
адреса, продажи, план по ТП, "нет накладных") в SQLite через Prisma, плюс
выгрузка сформированного заказа в шаблон 1С. Реализует задачу [PHA-73].

Стек: Node.js (без фреймворка) + Prisma + SQLite, `xlsx`/`jszip` для чтения
Excel — соответствует прямому указанию заказчика по стеку (см. `## Область
охвата и известные ограничения` ниже).

## Быстрый старт

```bash
cd etl-loader
npm install
cp .env.example .env        # DATABASE_URL="file:./dev.db"
npx prisma migrate deploy   # применяет миграции к dev.db (уже есть в prisma/migrations)

# Загрузить образцы файлов из PHA-64 (уже лежат в samples/) в dev.db:
node src/cli.js load ./samples ./out

# Демонстрация сценария "формирование заказа -> выгрузка в шаблон 1С"
# на реально загруженных данных (берёт первого клиента со скидкой):
node src/cli.js demo ./out/demo_order.xlsx
```

`npm run load` / `npm run demo` — то же самое через package.json-скрипты.

Открыть данные глазами: `npx prisma studio`.

## Веб-приложение (каркас + экран "Главная")

Реализует [PHA-75]: минимальный Express-сервер поверх той же базы (Prisma/
SQLite), с одним экраном — мобильный дашборд "Главная".

```bash
npm install
cp .env.example .env
npx prisma migrate deploy
npm run load                # наполнить dev.db реальными данными из samples/
PORT=3000 npm start         # сервер слушает process.env.PORT (по умолчанию 3000)
```

Открыть `http://localhost:3000/` — экран "Главная" (товарооборот за месяц и
план, оплачиваемые задачи с весом в бонусе, просроченная задолженность).
Все цифры читаются из БД через Prisma, ничего не захардкожено.

Структура (задел под следующие экраны — клиенты/заказы/заметки/фото/гео):

```
src/web/
  app.js               — сборка Express-приложения (view engine, static, роуты)
  routes/home.js        — GET / -> экран "Главная"
  services/dashboardService.js — Prisma-запросы для дашборда
  views/home.ejs         — вёрстка экрана (мобильная, EJS)
  public/styles.css      — стили
src/server.js            — точка входа, слушает PORT
```

Пока нет экрана входа, дашборд показывает данные одного фиксированного ТП
(`PLACEHOLDER_MANAGER_ID` в `dashboardService.js`) — это явное допущение,
см. комментарий в коде; экран логина/выбора пользователя вне рамок PHA-75.

## Что загружается и куда

| Источник (файл) | Парсер | Таблицы |
|---|---|---|
| Прайс-лист с остатками | `src/parsers/priceList.js` | `Product`, `Stock` |
| Действующие скидки | `src/parsers/discounts.js` | `Discount` (per-client + region default) |
| ДЗ и просроченная задолженность | `src/parsers/debt.js` | `Debt` |
| Адреса доставки с координатами | `src/parsers/addresses.js` | `Region`, `Manager`, `Client`, `ClientAddress` |
| Нет накладных на отгрузку | `src/parsers/missingInvoices.js` | `MissingInvoice` |
| Продажи (регион→менеджер→клиент→бренд) | `src/parsers/salesFacts.js` | `SalesFact` |
| Загрузка по ТП (план, 3 листа) | `src/parsers/plan.js` | `Plan`, `AcbPlan` |
| Шаблон загрузки заказа | `src/exportOrderTemplate.js` | (только экспорт, не источник) |

Каждый источник обрабатывается независимо: `src/lib/validation.js` собирает
построчные ошибки, не прерывая загрузку остальных строк (ТЗ 6.12 — "отчёт
об ошибках/пропусках до применения"). После каждой загрузки в таблицу
`DataLoad` пишется сводка (файл, кол-во строк, статус), а в
`ValidationError` — конкретные проблемы с номером строки/полем/значением.
`node src/cli.js load` также пишет `out/load_report.json` со сводкой по всем
источникам за прогон.

Файлы находятся по контенту имени (`src/lib/findSourceFile.js`), а не по
точному совпадению — 1С каждый день меняет имя файла плана (`..._ИЮЛЬ_2026`
→ `..._АВГУСТ_2026`), поэтому точное имя было бы хрупким.

## Бизнес-логика скидок и цены клиента (`src/pricing.js`)

- Приоритет скидки: индивидуальная скидка клиента → базовая скидка региона
  → без скидки (ТЗ 6.6). Истёкшие скидки (`Дата окончания скидки` < сегодня)
  игнорируются.
- Цена клиента считается от уже сниженной прайсом «Цены» (не от «Цены без
  скидки»): раздел 4.1 ТЗ прямо говорит "клиентская скидка применяется
  поверх" сниженной цены — это и взято как рабочая гипотеза, но раздел 7.1
  прямо помечает оба варианта как открытый вопрос к заказчику.
- ⚑ **"Цена поставщика"** для шаблона выгрузки заказа берётся как «Цена без
  скидки» из прайса — в прайсе нет отдельной колонки себестоимости (ТЗ 7.1,
  открытый вопрос #2). Нужно заменить на реальную себестоимость после
  согласования.

## Область охвата и известные ограничения

**Почему схема БД написана с нуля, а не переиспользует модели PHA-68.**
Задача явно просит переиспользовать модели MVP-приложения (PHA-68), но на
момент выполнения к проекту не был привязан ни один resource
(`multica project resource list` — пусто), а код PHA-68 существовал только
в рабочей директории той задачи, которая недоступна отсюда — это тот же
блокер, что PM и QA независимо фиксировали в комментариях к
[PHA-73]/[PHA-70] ("репозиторий не привязан... QA не может ни git clone, ни
docker compose up"). Рабочая директория этого прогона тоже оказалась
чистой (не git-репозиторий, без кода PHA-68) — предыдущая сессия по этой же
задаче, судя по её комментарию "kicked off the final ETL load into the real
dev.db", работала в другой, тоже не сохранившейся директории. Схема в
`prisma/schema.prisma` покрывает только сущности, нужные самому загрузчику
(ТЗ раздел 5, без Order/GPS/Фото-аудита/Заметок — это зона MVP-приложения).
**Как только репозиторий PHA-68 станет доступен, эти модели надо свести с
уже существующими, а не держать как отдельный источник истины** — сейчас
они сознательно разделены, чтобы не гадать вслепую.

**Full-replace семантика вместо версионирования.** Каждая загрузка полностью
заменяет содержимое соответствующих таблиц (`deleteMany` + вставка), кроме
`DataLoad`/`ValidationError`, которые копятся как история запусков. ТЗ 6.12
просит "версионирование загрузок и откат к предыдущей версии" — это
отдельная, более крупная фича (нужно хранить снапшоты, не только факт
последнего запуска), не реализована здесь: за пределами задачи "загрузчик
ETL", это скорее относится к MVP-приложению/админке.

**Иерархия строк в "ДЗ" и "Продажи" — как определяется.** Оба файла — это
плоские 1C-выгрузки иерархических отчётов, где вложенность (регион →
менеджер → клиент → бренд) не показана отступами, а закодирована в
атрибуте `outlineLevel` каждой строки `<row>` в XML (`src/lib/outline.js`
читает это напрямую из xlsx-архива, так как `xlsx`/SheetJS не отдаёт эту
метадату через `sheet_to_json`).

- В файле **ДЗ** это работает идеально (0 ошибок на реальных данных):
  уровень 0 = регион, 1 = менеджер, 2 = клиент/партнёр, 3 = договор,
  4 = заказ. Берётся только строка уровня 2 (клиент) — это уже готовый
  роллап 1С по всем его договорам/заказам, что и требует ТЗ 4.3.
- В файле **Продажи** `outlineLevel` оказался ненадёжен: у ~99% клиентских
  блоков (2918 из 2946 в примере) 1С вообще не выводит отдельную строку с
  именем клиента — граница между клиентами схлопывается на тот же уровень,
  что и обычные строки бренда. Поэтому классификация строк в
  `salesFacts.js` **не использует уровень для различения клиент/бренд**, а
  использует частоту повторения текста строки по всему файлу: настоящий
  бренд (BOSCH, WINKOD...) покупают десятки-сотни разных клиентов, поэтому
  его название повторяется много раз; имя клиента по построению уникально и
  встречается ровно один раз. Порог `BRAND_FREQUENCY_THRESHOLD = 2` на
  примере файла режет очень чисto: 3026 меток с частотой 1 (менеджеры,
  регион, клиенты) против 137 меток с частотой ≥ 2 (это практически
  совпадает с известным каталогом брендов, ~98-187 названий). Менеджер
  определяется не частотой, а сверкой с ростером из файла плана (шесть
  ФИО) — единственный надёжный список менеджеров, который у нас есть.
  Подробности и обоснование — комментарии в начале `src/parsers/salesFacts.js`.
  **Ограничение:** бренд, который какой-то клиент купил всего 1 раз за все
  19 месяцев, будет по ошибке принят за новую границу клиента, что рвёт
  историю этого клиента на две записи. На реальном файле такое случилось
  лишь единожды (залогировано как `Инфо:`-заметка), но это не гарантия —
  корень проблемы в том, что данные в этом конкретном экспорте 1С реально
  не содержат достаточно информации, чтобы решить эту задачу со 100%
  точностью для всех строк; более чистая выгрузка (с явной колонкой кода
  клиента, как в файле адресов) сняла бы вопрос полностью.

**Сопоставление клиентов между файлами — по имени, не по коду.** Только
файл адресов несёт "Код клиента" (реальный первичный ключ 1С). Остальные
файлы (скидки/ДЗ/продажи/нет-накладных) ссылаются на клиента только по
названию, поэтому загрузчик сопоставляет их по точному совпадению строки
имени (после `trim()`), с созданием клиента-заглушки (без кода/адреса), если
имя не встретилось в файле адресов — это логируется как `Инфо:`-заметка,
не как ошибка. Два разных реальных клиента с абсолютно одинаковым названием
(без кода это неразличимо) будут слиты в одну запись — это тоже
принципиальное ограничение источников, а не то, что можно починить в коде
загрузчика.

**Единственный регион.** Пилотные файлы содержат один регион (Алматы, ТЗ
3.5) — `defaultRegionId` в `src/load.js` берётся из первой строки файла
адресов и используется для всех менеджеров без региона в других файлах.
Мульти-регион (Астана, Кар-Сити) потребует, чтобы каждый источник указывал
регион явно (сейчас файл "Продажи" вычисляет регион из своей строки
"Отбор:", но остальные файлы этого не делают).

## Тесты и проверка

Готового набора unit-тестов нет (`npm test` — заглушка); проверка сделана
прогоном загрузчика на реальных образцах вложений из [PHA-64] (см. таблицу
выше) и ручной проверкой сквозного сценария "клиент → скидка → цена →
выгрузка в шаблон заказа" через `node src/cli.js demo`.

## Структура

```
prisma/schema.prisma   — модель данных (SQLite)
src/lib/               — общие хелперы (чтение xlsx, outlineLevel, числа/даты/координаты, сбор ошибок)
src/parsers/           — по одному парсеру на файл-источник
src/pricing.js         — скидка + цена клиента (ТЗ 6.6/7.1)
src/exportOrderTemplate.js — выгрузка в шаблон 1С (ТЗ 4.8/6.5)
src/load.js            — оркестрация: parse -> Prisma -> DataLoad/ValidationError
src/cli.js             — `load` / `demo`
samples/               — образцы вложений из PHA-64 (для повторной проверки)
```
